# Инструкция по развёртыванию и использованию ArchAssistant

ArchAssistant — это персональный помощник для генерации Java/Kotlin-кода с контролем архитектурных стандартов. Система состоит из двух частей:

1. **Backend** на Kotlin/Spring Boot/Spring AI, который хранит правила проекта, формирует контекст, запускает генерацию, валидирует код через ArchUnit, рассчитывает метрику **ArchitectureComplianceScore**, работает с overlay-контекстом и метриками.
2. **VS Code Extension** на TypeScript, через которое разработчик запускает backend, выбирает проект, редактирует архитектурные правила, отправляет запросы на генерацию, сохраняет файлы, добавляет их в контекст и просматривает метрики.

---

## 1. Предварительные требования

Перед запуском убедитесь, что установлены:

* **Docker Desktop** или Docker Engine с поддержкой `docker compose`
* **Visual Studio Code**
* **Node.js** (для сборки расширения)
* **JDK 21** или совместимая версия, если планируется локальная сборка backend без Docker
* **Git**

Также необходимо подготовить доступ к одной из LLM-моделей, поддерживаемых backend:

* **Mistral**
* **GigaChat**

---

## 2. Структура проекта

В репозитории обычно используются следующие основные части:

* `archassistant-backend/` — backend-сервис
* `archassistant-vscode/` или аналогичная папка — исходники расширения VS Code
* `docker-compose.yml` — запуск PostgreSQL и backend в контейнерах
* `.env` — переменные окружения для Docker и моделей
* `rules-configurations/` — YAML-конфигурации правил для проектов

---

## 3. Настройка `.env`

В корне репозитория должен быть файл `.env`. Он используется `docker-compose.yml` и backend-контейнерами.

Пример содержимого:

```env
POSTGRES_DB=archassistant
POSTGRES_USER=postgres
POSTGRES_PASSWORD=postgres

SPRING_PROFILES_ACTIVE=prod
LLM_PROVIDER=mistral

MISTRAL_BASE_URL=https://api.mistral.ai
MISTRAL_API_KEY=YOUR_MISTRAL_API_KEY
MISTRAL_MODEL=mistral-medium-latest
MISTRAL_TEMPERATURE=0.2
MISTRAL_MAX_TOKENS=10000

GIGACHAT_BASE_URL=https://gigachat.devices.sberbank.ru/api/v1
GIGACHAT_API_KEY=YOUR_GIGACHAT_API_KEY
GIGACHAT_SCOPE=GIGACHAT_API_PERS
GIGACHAT_MODEL=GigaChat
GIGACHAT_TEMPERATURE=0.2
GIGACHAT_MAX_TOKENS=10000
GIGACHAT_CONNECT_TIMEOUT=15s
GIGACHAT_READ_TIMEOUT=30s
GIGACHAT_UNSAFE_SSL=false
```

### Что означает каждая группа переменных

* `POSTGRES_*` — параметры базы данных.
* `SPRING_PROFILES_ACTIVE` — активный профиль Spring.
* `LLM_PROVIDER` — активный провайдер модели (`mistral` или `gigachat`).
* `MISTRAL_*` — настройки Mistral.
* `GIGACHAT_*` — настройки GigaChat.

Если планируется работа только с одной моделью, достаточно заполнить только её ключи. Остальные можно оставить пустыми.

## 4. Развёртывание через Docker (необязательный этап)

При нормальной работе все docker-контейнеры запускаются самим расширением VS Code, но перед первым использованием рекомендуется протестировать возможность запуска серверной части.

### 4.1. Подготовьте путь к проекту

Backend ожидает, что в контейнер будет смонтирован каталог анализируемого Java/Kotlin-проекта. Этот путь передаётся через переменную окружения `PROJECT_PATH`.

Пример:

```bash
export PROJECT_PATH="C:\\Users\\Kirill\\Projects\\my-spring-project"
```

Если используете PowerShell:

```powershell
$env:PROJECT_PATH = "C:\Users\Kirill\Projects\my-spring-project"
```

Важно: в выбранном проекте должны быть доступны **скомпилированные классы**. Для Gradle-проектов это обычно:

* `build/classes/kotlin/main`
* `build/classes/java/main`
* `build/resources/main`

Для Maven-проектов:

* `target/classes`

Для IntelliJ IDEA-проектов:

* `out/production/<project-name>`

### 4.2. Проверьте docker-compose

В корне репозитория находится `docker-compose.yml`. Он поднимает:

* PostgreSQL
* backend-сервис ArchAssistant

Backend получает путь к проекту в режиме **read-only** и не изменяет исходники напрямую.

### 4.3. Запустите контейнеры

Из корня репозитория выполните:

```bash
docker compose --env-file .env up -d --build
```

Если у вас старая версия Docker Compose:

```bash
docker-compose --env-file .env up -d --build
```

Если хотите поднять только базу данных и backend в одном сценарии, используйте стандартный compose-файл без изменений.

### 4.4. Проверьте состояние сервисов

Посмотреть контейнеры:

```bash
docker compose ps
```

Посмотреть логи backend:

```bash
docker compose logs -f backend
```

Посмотреть логи PostgreSQL:

```bash
docker compose logs -f postgres
```

### 4.5. Проверка backend

Когда backend запустится, его health endpoint должен вернуть статус `UP`.

Проверка в браузере или через curl:

```bash
curl http://localhost:8080/api/health
```

Ожидаемый ответ — JSON со статусом `UP`, версией и timestamp.

Если backend не отвечает:

* проверьте переменную `PROJECT_PATH`
* проверьте заполнение API-ключей моделей
* убедитесь, что Docker может читать каталог проекта
* откройте логи backend

## 6. Настройки расширения VS Code

Расширение использует следующие пользовательские настройки:

* `archassistant.backendUrl` — адрес backend API, по умолчанию `http://localhost:8080`
* `archassistant.composeDirectory` — папка, в которой лежит `docker-compose.yml`
* `archassistant.composeServiceName` — имя сервиса backend в compose, по умолчанию `backend`
* `archassistant.autoStartBackend` — автоматически запускать backend при старте проекта

Открыть настройки можно через:

* `Ctrl+,` → поиск `ArchAssistant`
* либо через JSON settings

Пример:

```json
{
  "archassistant.backendUrl": "http://localhost:8080",
  "archassistant.composeDirectory": "C:/Users/Kirill/archassistant",
  "archassistant.composeServiceName": "backend",
  "archassistant.autoStartBackend": true
}
```

## 7. Первый запуск ArchAssistant

После установки расширения в боковой панели VS Code появится контейнер **ArchAssistant**.

### Шаг 1. Нажмите `Start Project`

Откроется окно выбора папки проекта.

Нужно выбрать **папку анализируемого Java/Kotlin-проекта**, в которой есть скомпилированные классы.

Если `autoStartBackend` включён, расширение:

1. определит `projectId` по имени папки;
2. найдёт папку с `docker-compose.yml`;
3. запустит контейнер backend;
4. дождётся статуса health `UP`;
5. сохранит проект в локальном состоянии расширения.

Во время запуска в правом нижнем углу VS Code показывается прогресс.

### Шаг 2. Нажмите `Configure Project`

После запуска проекта необходимо выполнить конфигурацию.

Команда:

* сохраняет путь проекта в backend;
* выполняет первичное сканирование структуры;
* запускает генерацию предложенных правил;
* сохраняет предложения в локальное состояние;
* обновляет дерево правил.

Если для проекта уже есть локальная YAML-конфигурация, расширение сначала восстановит её, а затем синхронизирует с backend.

## 8. Работа с архитектурными правилами

В панели **Rules** доступны две группы:

* **Saved rules** — сохранённые правила проекта
* **Suggested rules** — правила, предложенные backend после анализа структуры

### 8.1. Кнопки Saved rules

* **Get Actual Rules** — загрузить текущую YAML-конфигурацию из backend
* **Save Rules** — отправить текущую конфигурацию на backend
* **Import YAML** — импортировать правила из локального YAML-файла
* **Export YAML** — сохранить текущую конфигурацию в YAML
* **Edit Rule** — редактировать правило
* **Delete Rule** — удалить правило

### 8.2. Кнопки Suggested rules

* **Add Custom Rule** — создать собственное правило вручную
* **Add Suggested Rule** — добавить предложенное правило в Saved
* **Refresh Suggestions** — повторно просканировать проект и обновить предложения

### 8.3. Тоггл правила

Для каждого сохранённого правила есть переключатель активности:

* включено — правило участвует в генерации и проверке
* выключено — правило хранится, но не применяется

## 9. Как устроена форма правила

Редактор правила открывается как отдельный Webview.

Общие поля есть почти у всех правил:

* `id`
* `name`
* `description`
* `type`
* `from_package`
* `constraint`
* `severity`
* `weight`
* `enabled`

Дополнительные поля появляются в зависимости от типа правила.

### 9.1. DEPENDENCY

Подходят для правил вида “не зависит от / должен зависеть от”.

Обычно используются:

* `from_package`
* `to_package` или `to_packages`
* иногда `from_selector_mode` и `to_selector_mode`

Примеры:

* domain не зависит от infrastructure
* controllers не зависят от repository

### 9.2. NAMING_CONVENTION

Используются для проверки суффикса или префикса именования.

Дополнительные поля:

* `pattern`

Примеры:

* `Controller`
* `Service`
* `Repository`
* `Resource`
* `Client`

### 9.3. ANNOTATION_CHECK

Подходят для проверки аннотаций.

Дополнительные поля:

* `annotation`

Примеры:

* `@RestController`
* `@Service`
* `@Repository`
* `@Entity`
* `@Configuration`

### 9.4. MODIFIER_CHECK

Используются для проверки модификаторов.

Дополнительные поля:

* ничего сверх базовых, но правило может требовать `public`, `abstract`, `final` и т.д. в зависимости от `constraint`

Примеры:

* controllers should be public
* repositories should be public

### 9.5. METHOD_SIGNATURE_CHECK

Используются для проверки сигнатуры метода.

Дополнительные поля:

* `from_method_name_pattern`
* `from_return_type`
* `from_parameter_types`

Примеры:

* `fallback()` возвращает `ResponseEntity`
* `getOwnerDetails(int)` возвращает `Mono`

### 9.6. FIELD_CHECK

Используются для проверки полей класса.

Дополнительные поля:

* `from_field_name_pattern`
* `from_field_type`
* `from_modifiers`

### 9.7. INTERFACE_CHECK

Используются для проверки реализации интерфейсов.

Дополнительные поля:

* `to_package`
* `to_name_pattern`

Примеры:

* repository implementation должен реализовывать domain repository contract

### 9.8. INHERITANCE_CHECK

Используются для проверки наследования.

Дополнительные поля:

* `to_package`
* `to_name_pattern`

Примеры:

* custom exceptions должны наследоваться от `RuntimeException`

### 9.9. CYCLE_CHECK

Используются для проверки циклов зависимостей между пакетами.

Дополнительные поля:

* `slice_pattern`
* `max_cycle_length`

## 10. Подтверждение и сохранение правила

При создании или редактировании правила:

1. откроется форма редактирования;
2. заполните обязательные поля;
3. нажмите **Save**;
4. если правило некорректно, будет показано сообщение об ошибке;
5. если всё в порядке, правило попадёт в Saved rules.

Если вы нажмёте **Cancel**, изменения не сохранятся.

## 11. Импорт и экспорт YAML

### 11.1. Импорт

Команда **Import YAML**:

1. открывает диалог выбора файла;
2. ожидает `.yaml` или `.yml`;
3. читает YAML;
4. валидирует структуру;
5. нормализует значения;
6. сохраняет в draft-конфигурацию;
7. отправляет её в backend.

Если YAML повреждён или структура неверна, появится сообщение об ошибке.

### 11.2. Экспорт

Команда **Export YAML**:

1. собирает текущую конфигурацию;
2. удаляет пустые поля;
3. формирует компактный YAML-файл;
4. предлагает выбрать место сохранения.

## 12. Генерация кода

Откройте панель **Generate Code** через:

* кнопку в Sidebar
* команду `Generate Code`

### Поля формы генерации

Обычно доступны:

* текст запроса (`prompt`)
* стратегия: `PRE`, `POST`, `HYBRID`
* ожидаемое имя класса
* максимальное число итераций

### Как работает генерация

1. расширение собирает `CodeGenerationRequest`;
2. отправляет запрос в backend;
3. backend генерирует код через LLM;
4. код проверяется через ArchUnit;
5. считается ArchitectureComplianceScore;
6. результат возвращается в Webview.

### Что показывается в результате

* итоговый score;
* число итераций;
* предупреждения;
* список нарушений;
* список сгенерированных файлов;
* текст кода.

### Кнопки результата

* **Open Files** — открыть сгенерированные файлы
* **Save Selected** — сохранить выбранные файлы в проект
* **Save & Add to Context** — сохранить и отправить файлы в overlay-контекст backend
* **Clear Project Context** — очистить overlay-контекст проекта на backend

---

## 13. Что происходит при кнопках файлов

### Save Selected

Файлы сохраняются в проект в папки:

* `src/main/java`
* `src/main/kotlin`

Расширение само определяет язык по коду и расширению файла.

После сохранения можно сразу открыть файл в редакторе.

### Save & Add to Context

Это двухэтапная операция:

1. файлы сохраняются локально;
2. затем отправляются в backend, где создаётся overlay-контекст.

Если операция успешна, последующие генерации учитывают новые классы и импорты.

### Clear Project Context

Удаляет overlay-контекст на backend и очищает временные данные, чтобы следующий запрос начинался без добавленных generated files.

## 14. Панель метрик

Команда **Show Metrics** открывает панель аналитики.

Панель отображает:

* сводку по проекту;
* сравнение стратегий `PRE`, `POST`, `HYBRID`;
* историю последних генераций;
* рекомендации по выбору стратегии;
* экспорт данных;
* очистку истории.

### Что делает `Compare Strategies`

Показывает:

* число генераций по каждой стратегии;
* процент успешных генераций;
* средний score;
* среднее число итераций;
* среднее время генерации, валидации и общее время;
* среднее число нарушений;
* рекомендацию с confidence.

### Что делает `Export Metrics`

Экспорт предлагает:

* формат: `CSV`, `JSON`, `JSON (Pretty)`
* включать ли violations
* экспортировать ли весь период или отфильтровать по датам

После выбора файла данные сохраняются локально.

### Что делает `Clear History`

Удаляет историю генераций конкретного проекта после подтверждения в диалоге.

## 15. Как реагировать на всплывающие окна

### При запуске проекта

* **Select folder** — выберите папку с проектом
* **Select folder with docker-compose.yml** — выберите папку, где лежит `docker-compose.yml`
* **Starting ArchAssistant backend** — подождите завершения запуска
* **Backend is not reachable** — проверьте Docker, `.env`, путь к проекту и логи backend

### При генерации

* **No project configured** — сначала нажмите `Start Project` или `Configure Project`
* **No files selected** — выберите хотя бы один файл перед сохранением
* **Failed to generate / save / sync** — проверьте backend-логи и валидность исходного кода

### При редактировании правил

* **Delete rule?** — подтвердите удаление, если правило больше не нужно
* **No suggested rules are available** — сначала нажмите Refresh Suggestions
* **Invalid YAML structure** — исправьте YAML-файл и импортируйте повторно

### При экспорте метрик

* выберите формат;
* выберите, включать ли violations;
* при необходимости задайте диапазон дат;
* затем выберите путь сохранения файла.

### При очистке истории

Появится подтверждение удаления всех записей по проекту. Нажимайте **Clear** только если действительно хотите очистить историю.

## 16. Полезный сценарий работы

Рекомендуемый порядок работы такой:

1. Запустить Docker и backend.
2. Открыть VS Code с расширением ArchAssistant.
3. Нажать **Start Project** и выбрать папку проекта.
4. Нажать **Configure Project**.
5. Проверить предложенные правила.
6. Отредактировать/добавить правила в панели **Rules**.
7. Нажать **Save Rules**.
8. Открыть **Generate Code** и отправить запрос.
9. Посмотреть score и violations.
10. Сохранить нужные файлы.
11. При необходимости добавить их в overlay-контекст.
12. Посмотреть **Metrics** и сравнить стратегии.
13. При необходимости экспортировать результаты.

## 18. Конфигурация правил проекта

Для каждого проекта используется YAML-файл с правилами. Он хранится отдельно от исходного кода и может импортироваться/экспортироваться через расширение.

Обычно конфигурация включает:

* `version`
* `project_id`
* `project_type`
* `settings`
* `rules`

Правила можно редактировать вручную, добавлять предложенные системой или формировать заново после повторного сканирования проекта.

## 20. Коротко о возможных проблемах

Если что-то не работает:

* проверьте, что `PROJECT_PATH` указывает на существующий проект
* убедитесь, что `docker-compose.yml` найден в указанной папке
* проверьте, что backend отвечает на `http://localhost:8080`
* проверьте ключи LLM
* убедитесь, что в проекте есть compiled classes
* посмотрите логи backend и сообщение в Output Channel `ArchAssistant`

## 21. Что делает каждый главный элемент интерфейса

### Sidebar → Project

Показывает:

* статус backend;
* текущий проект;
* количество правил;
* статус компиляции проекта;
* основные кнопки запуска и генерации.

### Sidebar → Rules

Показывает:

* Saved rules;
* Suggested rules;
* команды импорта/экспорта YAML;
* редактирование, удаление, добавление правил;
* переключатель enabled.

### Generate Code panel

Показывает:

* форму запроса;
* выбранную стратегию;
* score;
* нарушения;
* код;
* список файлов;
* действия по сохранению и синхронизации.

### Metrics panel

Показывает:

* сводку по проекту;
* сравнение стратегий;
* историю генераций;
* экспорт;
* очистку истории.

---

## 22. Итог

ArchAssistant позволяет:

* запускать backend автоматически через VS Code;
* настраивать архитектурные правила проекта;
* генерировать код через LLM с учётом правил;
* проверять код через ArchUnit;
* сохранять и синхронизировать generated files;
* собирать и экспортировать метрики;
* сравнивать стратегии `PRE`, `POST`, `HYBRID`.

Это README можно использовать как инструкцию для развёртывания, демонстрации и повторного запуска системы.
